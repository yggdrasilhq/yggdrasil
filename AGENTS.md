# AGENTS

## Mission

Build and release dual-profile Debian live ISOs (`server` and `kde`) with ZFS + LXC defaults.

## Release Rules

- Always build both profiles.
- Never ship if smoke tests fail for either profile.
- Keep profile-specific ISO retention policy:
  - latest ISO from last 3 days
  - last 2 older releases

## Build Entrypoints

- `./mkconfig.sh --profile both`
- `ygg-cli` (external TUI repo)
- `./tests/smoke/run.sh`

## Testing Expectations

Smoke tests must cover, at minimum:

- `ygg-import-zpool-at-boot.service`
- `ygg-lxc-autostart.service`
- `ygg-infisical-ensure.service`
- `/etc/lxc/lxc.conf` and `/etc/lxc/default.conf` semantics
- KDE profile post-boot `zfs` userland check

## Public Docs

Documentation lives in the separate `ygg-docs` repository and is published to your docs host.

## README And Release Notes Contract

- Treat `README.md` as the primary user-facing manual, product pitch, and first proof of software quality.
- Treat the changelog (`CHANGELOG.md` or the repo's established changelog path) as the second marketing surface and the source of curated release notes.
- When user-visible behavior changes, update the README and changelog in the same workstream instead of letting them drift.
- Release pages should reuse the relevant curated changelog entry rather than autogenerated notes whenever the forge or hosting platform supports custom release bodies.
- Keep examples concrete, user-facing, and pasteable; avoid release notes that only describe internal refactors.
