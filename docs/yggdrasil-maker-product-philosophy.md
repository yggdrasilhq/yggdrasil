# Design: yggdrasil-maker

Generated by /office-hours on 2026-04-02
Branch: main
Repo: local/yggdrasil
Status: DRAFT
Mode: Builder

## Problem Statement

`yggdrasil` currently has a split personality.

The real build truth lives here, in native config files, `./mkconfig.sh`, profile-specific build hooks, and smoke tests. But the guided user experience lives somewhere else, in the external `yggcli` repo. That split made sense as an experiment. It is now the wrong product shape.

The operator wants one front door, one source tree, one release story, and one mental model:

- this repo defines how Yggdrasil server and KDE ISOs are built
- this repo should also contain the best supported way to configure and launch those builds
- the guided path should feel like a real Yggdrasil app, not a detached wrapper that happens to shell out here

The proposed change is to decommission the external `yggcli` repo and replace it with an in-repo Rust + Dioxus app, `yggdrasil-maker`, shaped like the `yggterm` family: an app crate plus internal support crates, with a real desktop UI and reusable shell primitives.

## What Makes This Cool

The cool part is not "GUI instead of TUI." That is too small.

The cool part is that the thing operators actually use to make the server becomes a first-class product surface:

- the build system and the guided UX stop drifting apart
- the same repo owns config truth, build truth, smoke truth, and release truth
- the app can explain what matters before boot, profile choice, ZFS and LXC defaults, Infisical mode, KDE-specific validation, and smoke status
- the app can look and behave like the rest of the Ygg desktop family instead of a one-off terminal wizard

Done right, `yggdrasil-maker` turns the repo from "build scripts plus optional wrapper" into "the actual product for making Yggdrasil hosts."

## Constraints

- The build engine already exists and must remain trustworthy: `./mkconfig.sh --profile both`, `./tests/smoke/run.sh`, and the profile hooks are the release-critical path.
- Both profiles remain mandatory. No product flow should normalize "server only unless you remember KDE later."
- Never ship when smoke fails for either profile.
- Native config files stay real and editable. The app is a front door, not a proprietary format trap.
- Public long-form docs remain in `ygg-docs`, while this repo keeps the concise operator build story.
- The current README still tells users to use `yggcli`; changing philosophy here will require a later README and release-note rewrite.
- Cross-platform UX does not imply cross-platform ISO building. Linux can build the ISO locally. Other platforms may need export-only or remote-builder flows.
- The design should feel aligned with `yggterm` crate patterns: app crate at the edge, reusable internal crates beneath it, Dioxus desktop shell, and UI primitives treated as product infrastructure rather than throwaway glue.

## Premises

1. The guided UX for making Yggdrasil hosts is part of the product, not an external accessory.
2. `yggcli` should be decommissioned because the split-repo boundary creates documentation drift, release drift, and ecosystem confusion.
3. `yggdrasil-maker` should write and validate the same native config that `mkconfig.sh` already accepts, not invent a second config model.
4. The app should orchestrate the existing build and smoke pipeline first, and only absorb deeper build logic after behavior is proven equivalent.
5. "All platforms" should mean the app can run everywhere, but the build workflow must truthfully expose which actions require a Linux builder.
6. Decommissioning `yggcli` is safe only after `yggdrasil-maker` reaches parity on the main operator path: configure, build both profiles, inspect artifacts, and surface smoke results clearly.

## Approaches Considered

### Approach A: Thin Maker Shell

Summary: Build `yggdrasil-maker` as an in-repo Dioxus desktop app that edits tracked config semantics, runs `mkconfig.sh`, and renders build plus smoke progress. Keep the shell scripts as the only build engine.

Effort: M
Risk: Low

Pros:

- smallest migration, because release-critical behavior stays in the existing shell pipeline
- preserves native TOML as the source of truth
- lets the app focus on UX, validation, progress, and operator confidence
- matches the immediate reason to kill `yggcli`, the front door belongs in this repo

Cons:

- some shell-driven behavior will remain awkward to introspect at first
- app UX quality depends on clean process boundaries and structured logs that do not fully exist yet
- cross-platform users may still need export-only or remote-build workflows until Linux-builder support is designed

Reuses:

- `ygg.example.toml`
- `mkconfig.sh`
- `tests/smoke/run.sh`
- existing profile hooks and artifact naming

### Approach B: Full Rust Build Core

Summary: Create `yggdrasil-maker` plus Rust domain crates that gradually replace TOML parsing, config validation, build orchestration, artifact retention, and smoke invocation. Shell scripts become thin compatibility shims or disappear over time.

Effort: XL
Risk: High

Pros:

- cleanest long-term architecture
- richest app introspection, because the UI can talk to typed domain logic instead of parsing shell output
- easier future support for remote builders, saved build sessions, and richer validation

Cons:

- large migration surface on release-critical code
- much higher risk of breaking dual-profile behavior and smoke expectations during the rewrite
- easy to confuse architecture cleanup with user value and spend months moving plumbing

Reuses:

- current profile semantics
- current smoke expectations
- release rules and retention rules

### Approach C: Cross-Platform Control Plane

Summary: Treat `yggdrasil-maker` as a cross-platform operator console. It owns config authoring, diffing, validation, and release visibility everywhere, but local ISO builds are Linux-only. Non-Linux clients either export config bundles or dispatch builds to a Linux builder host.

Effort: L
Risk: Medium

Pros:

- honest about platform reality instead of pretending Windows or macOS can build Debian live ISOs locally
- gives "all platforms" real value on day one
- creates a path toward remote-build UX without forcing a shell rewrite first

Cons:

- needs a clear concept of local build versus remote build from the start
- requires more product design than Approach A
- can become an infrastructure project if builder-host transport is pulled in too early

Reuses:

- existing config model
- existing build and smoke scripts on the Linux builder side
- future Ygg app-shell patterns inspired by `yggterm`

## Recommended Approach

Choose Approach A first, with Approach C as the explicit destination.

That means:

- ship an in-repo Dioxus app quickly
- keep `mkconfig.sh` and smoke scripts as the execution truth
- make native config editing and dual-profile visibility the first-class UX
- tell the truth about platform limits
- add remote-builder support later instead of pretending local cross-platform ISO building is real

This is the right sequence because the user pain is not "the shell scripts are written in bash." The pain is that the best supported way to use the product lives in the wrong repo and has the wrong feel.

Fix the front door first. Do not rewrite the foundation just because Rust is available.

## Product Philosophy

`yggdrasil-maker` should follow these rules:

1. One repo, one story.
   The repo that owns ISO behavior also owns the best guided UX for producing those ISOs.

2. Native config is sacred.
   The app may generate, edit, validate, and explain `ygg.local.toml`, but it does not hide or replace it.

3. Build both is the default mindset.
   The product should visually center the dual-profile release contract, not bury KDE as an afterthought.

4. Smoke truth is product truth.
   A build is not "done" when the ISO file exists. It is done when the relevant smoke checks pass and the operator can see that plainly.

5. UX should teach operations, not conceal them.
   ZFS import behavior, LXC defaults, Infisical mode, SSH key embedding, and profile differences should be understandable from the app.

6. Cross-platform means plan anywhere, build where it is honest.
   The app runs anywhere. The ISO builds where the platform and toolchain actually support it.

7. Match the Ygg app family.
   The crate structure, shell chrome, and interaction quality should feel like a sibling of `yggterm`, not a random admin panel.

## Open Questions

- Is v1 Linux-desktop-first, or do we require config-authoring support on macOS and Windows immediately?
- Should `yggdrasil-maker` launch the build directly, or first export a reviewed config plus exact command preview?
- Do we want to vendor or copy shared `yggui` patterns, or keep the alignment conceptual and repo-local?
- Is remote-builder support a v1 requirement for "all platforms," or a clearly labeled v2?
- Should ISO releases and app releases share one version stream, or should the app have its own versioning inside the repo?

## Success Criteria

- A new operator can discover the server versus KDE choice without reading raw scripts first.
- A user can author or edit a valid native config from the app and still inspect the file directly afterward.
- The app can run the supported build path for both profiles and surface smoke results clearly.
- The README can truthfully replace `yggcli` with `yggdrasil-maker` as the guided entrypoint.
- The external `yggcli` repo can be frozen without removing a capability users still need.

## Distribution Plan

- Deliver `yggdrasil-maker` as cross-platform desktop binaries through GitHub Releases.
- Keep ISO artifacts released through the existing Yggdrasil release flow, with curated release notes.
- In v1, support local build execution only on Linux. On non-Linux platforms, support config authoring, validation, and command export. Remote-builder support can follow.
- CI should build the desktop app for supported platforms and separately preserve the current Linux ISO build and smoke gates.

## Next Steps

1. Write the README-level story first: `yggdrasil-maker` is the new guided front door, `mkconfig.sh` remains the native direct path, `yggcli` moves to decommissioned status.
2. Define the app boundary: config editor, profile chooser, validation, build launcher, smoke viewer, artifact view.
3. Mirror the `yggterm` shape in this repo: workspace root, `apps/yggdrasil-maker`, and a small set of supporting crates.
4. Add structured build and smoke output that the app can consume without fragile log scraping.
5. Only after parity, mark the external `yggcli` repo read-only and point users here.
